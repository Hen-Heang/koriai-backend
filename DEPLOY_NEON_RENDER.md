# Deploying koriai-backend to Neon + Render

This app is a Spring Boot (Java 17) service built from the repo's `Dockerfile`.
- **Neon** hosts the Postgres database.
- **Render** hosts the application (Docker web service).

Flyway runs migrations automatically on startup, so you do **not** need to create
tables manually — just provide an empty Neon database.

---

## 1. Create the database on Neon

1. Sign up at https://neon.tech and create a **Project** (pick a region, e.g. Singapore).
2. Neon creates a database (default name `neondb`) and a role. Open **Connection Details**.
3. Copy the connection string. Neon gives you something like:

   ```
   postgresql://myuser:mypassword@ep-cool-name-123456-pooler.ap-southeast-1.aws.neon.tech/neondb?sslmode=require
   ```

4. Convert it into the three values Render needs. **Note the `jdbc:` prefix on the URL**
   and that the username/password are passed separately (not in the URL):

   | Render env var | Value (from the Neon string above) |
   |----------------|------------------------------------|
   | `DB_URL`       | `jdbc:postgresql://ep-cool-name-123456-pooler.ap-southeast-1.aws.neon.tech/neondb?sslmode=require` |
   | `DB_USERNAME`  | `myuser` |
   | `DB_PASSWORD`  | `mypassword` |

   - Keep `?sslmode=require` — Neon rejects non-SSL connections.
   - The `-pooler` host is fine (and recommended). It uses Neon's connection pooler.

> Neon free tier auto-suspends after inactivity. The first request after a pause may
> take a few seconds while the DB wakes up — this is normal.

---

## 2. Deploy the app on Render

You can deploy in two ways. The Blueprint (`render.yaml`) is the simplest.

### Option A — Blueprint (recommended)
1. Push this repo (including `render.yaml`) to GitHub.
2. In Render: **New → Blueprint**, connect the repo. Render reads `render.yaml`
   and creates the `koriai-backend` web service.
3. When prompted, fill in the env vars marked `sync: false`:
   - `DB_URL`, `DB_USERNAME`, `DB_PASSWORD` — from step 1.
   - `OPENAI_API_KEY` — your OpenAI key.
   - `CORS_ALLOWED_ORIGINS` — your frontend origin(s), comma-separated,
     e.g. `https://your-frontend.vercel.app`.
   - `JWT_SECRET` is generated automatically.
4. Click **Apply**. Render builds the Dockerfile and deploys.

### Option B — Manual web service
1. Render: **New → Web Service**, connect the repo.
2. Set **Runtime = Docker** (Render auto-detects the `Dockerfile`).
3. Set **Health Check Path** = `/api/health`.
4. Add the same env vars listed above under **Environment**.
5. Create the service.

---

## 3. Verify

Once the deploy goes green, hit the health endpoint:

```
https://<your-service>.onrender.com/api/health
```

You should get a JSON response with status `UP`. Check the Render **Logs** tab to
confirm Flyway ran the `V1..V12` migrations against Neon on first boot.

---

## Environment variables reference

| Variable                            | Required | Notes |
|-------------------------------------|----------|-------|
| `DB_URL`                            | yes      | JDBC URL incl. `?sslmode=require` |
| `DB_USERNAME`                       | yes      | Neon role |
| `DB_PASSWORD`                       | yes      | Neon password |
| `OPENAI_API_KEY`                    | yes      | OpenAI key (or set `OPENAI_MOCK_ENABLED=true` to mock) |
| `JWT_SECRET`                        | yes      | Long random secret (auto-generated by Blueprint) |
| `CORS_ALLOWED_ORIGINS`              | yes      | Comma-separated allowed frontend origins |
| `PORT`                              | no       | Render sets this automatically; the app binds to it |
| `SPRING_FLYWAY_BASELINE_ON_MIGRATE` | no       | Set `true` only if the Neon DB already has non-Flyway tables |
